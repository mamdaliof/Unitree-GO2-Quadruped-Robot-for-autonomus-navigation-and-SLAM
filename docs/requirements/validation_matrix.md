# System Validation & Verification Matrix

This document defines the validation methods used to verify each requirement from the `[[docs/requirements/system_requirements|System Requirements Specification]]`.

---

## 1. Verification Methods (VM)

We define six primary verification methods to validate software and hardware behaviors:

### VM-01: Virtual Mock & Optional Simulation Testing
* Running compiled nodes in a virtual mock environment using synthetic sensor message publishers (synthetic IMU, LiDAR coordinates, battery states) to verify state machine transitions, node logic, and ROS 2 topic interfaces. 
* *Optional Step:* Executing control, planning, and state machine modules in a full 3D physics-based simulation (e.g., Gazebo or Webots) if a matching robot model configuration is available.

### VM-02: Qualitative Map Consistency Check (Offline Bag Replay)
* Replaying pre-recorded sensor datasets through SLAM and perception nodes. Map quality and consistency are evaluated by measuring wall thickness (checking for double-walls or ghosting artifacts), corner sharpness, and drift alignment when returning to the starting node.

### VM-03: Physical Hardware Testing
* Deploying compiled nodes onto the physical Go2 Edu robot in a spacious, supervised test area (Cube, basement, or AMC) under safe-mode constraints (velocity $< 0.3$ m/s, human supervisor with manual override).

### VM-04: Software Fault Injection & API Audit
* Verifying code robustness and API alignment:
  * **Software Fault Injection:** Testing safety fallback actions (e.g., REQ-SFT-002 watchdog timeout, sensor dropouts, communication lag) safely in software by programmatically stopping publishers or injecting high-latency values, checking that the system transitions to safe states (`StopMove` or Idle SDK commands) in logs without physical collision risk.
  * **Static Audit:** Running static code analysis, compiler warning checks, unit tests (gtest/gmock/unittest), and verifying ROS 2 topic/message schemas.

### VM-05: Quantitative Loop Drift Check
* Measuring SLAM odometry drift by placing the robot at a precise physical start marker, commanding it to navigate a closed-loop path (e.g., 10 meters), returning it manually or autonomously to the exact same start position, and calculating the SLAM coordinate displacement error relative to the origin $(0,0,0)$.

### VM-06: Quantitative Point-to-Point Distance Check
* Marking two physical points on the floor separated by a known, manually-measured ground-truth distance (e.g., 5.0 meters). We walk the robot between these points and compare the distance calculated between the corresponding coordinate positions in the SLAM map against the physical reference.
